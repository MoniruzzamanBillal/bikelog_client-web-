The auth/token handling (`lib/tokenManager.ts`, axios interceptors) and the API-calling/mutation layer (`hooks/useApi.ts`, `utils/api.ts`) were both rebuilt from scratch in spec 01/02 rather than kept from the `table` scaffold's versions — see `architecture.md`'s "Auth & Access Model" and "API Calling & Mutation Pattern" for the exact shapes. On top of that, the session model enforces a **hard navigation gate**: no route besides `/login`/`/register` renders without a valid session, checked synchronously at `app/(main)/layout.tsx` (not left to an API call failing first) — see `architecture.md` Invariant 7.

The backend has two roles (`user`/`admin`) but no role-based authorization implemented yet. v1 of this frontend builds the `user`-role experience only, under `app/(main)/`; `app/(admin)/` is reserved (no files) for an admin panel later — see `project-overview.md`'s "Roles" section and `architecture.md` Invariant 8.

## Known Gaps / Open Questions

- **Auth cookie is not httpOnly.** `bikelog_server` returns the JWT in the JSON body and never sets cookies itself, so the client sets a plain (non-httpOnly) `cookies-next` cookie after login, via `lib/tokenManager.ts`. This is a deliberate, documented tradeoff (see `ai-workflow-rules.md`'s Protected Files section) — acceptable for a single-user personal tool, would need revisiting before this app ever had untrusted multi-tenant users.
- **No refresh-token flow exists, by design** — `bikelog_server` has no refresh-token endpoint at all, so there's no refresh-queue/retry logic to build. An expired token just means the interceptor redirects to `/login` — re-login is the only path back.
- **Session gate is a plain cookie check, not a `GET /auth/me` round-trip, by design** — `app/(main)/layout.tsx` only checks `getToken()` presence/expiry synchronously; it doesn't verify the token against the backend before rendering. A stale-but-not-yet-expired token would pass the gate and only fail on the first real API call (handled by the response interceptor's 401 branch). Accepted tradeoff for build speed — see `architecture.md`'s Auth & Access Model.
- **`useApi.ts`'s mutation hooks take the URL per-call, not per-hook, by design** — `usePost()`/`usePatch()`/`useDelete()` are generic (`mutateAsync({url, payload})`) rather than the more common "one hook per endpoint" shape. Domain hook files (`useBikes.ts` etc.) are what give call sites a fixed-endpoint convenience wrapper — see `architecture.md`'s example. `apiPut`/an id-in-URL/FormData-aware PATCH variant are not ported — `bikelog_server` has no `PUT` routes and that extra complexity isn't needed here.
- **`app/(admin)/` is empty and reserved, not a placeholder to fill in speculatively** — no admin spec exists yet. See `project-overview.md`'s "Roles" section.
- **Component bugs found in the `table`-scaffold inherited component library — fixed or avoided by design in spec 01:**
  - `hooks/useDebounce.ts` and `useDebounce.tsx` were byte-identical duplicates — the `.tsx` one is deleted (fixed in spec 01).
  - Several shared components (`TableFilter`, `TablePagination`, `DateSelect`, `BaseModal`) reference Tailwind classes (`table-border`, `primary-500`, `primary-600`) that didn't exist in `globals.css`'s `@theme` block — added in spec 01 (`--color-table-border`, `--color-primary-500`, `--color-primary-600`, all mapped onto existing tokens rather than a new hue).
  - `ControlledMultiSelectField`'s `onCreateOption` has a type bug (pushes a `Number` into a `string[]` field, no `undefined` guard) — Bike Log avoids this entirely by not using creatable multi-select anywhere (`partsReplaced` uses a plain text input instead, per spec 07). Not fixed, just avoided — the component itself still has the bug if some future spec reaches for it.
  - `PageHeader`/`StaticPageBreadcrumb` disagree on the dashboard route (`/dashboard` vs `/overview`) — standardized on `/dashboard` in this app's usage. Not yet fixed in the components themselves (no page uses either component yet); revisit whichever spec first wires up page headers/breadcrumbs.
